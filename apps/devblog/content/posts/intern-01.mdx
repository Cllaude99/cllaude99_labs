요즘 제가 인턴으로 일하면서 가장 많이 고민하고 있는 건 "어떻게 하면 효율적으로 업무를 처리할 수 있을까?" 에요.

입사하고 한달 정도의 기간동안 많은 양의 레거시 코드 및 구조를 파악하는데 소요한 것 같아요.
지금상황에서 이 컴포넌트를 사용하는게 맞을까? 이 로직은 어떻게 동작하는거지? 에 대한 고민도 많았고요.
이럴때마다 동료 개발자분들께 여쭈어보거나 AI를 활용해가며 조금씩 답을 찾아갈 수 있었어요.

시간이 어느덧 지나 3달차가 되는 지금은 반복되는 업무 속에서 조금씩 나만의 패턴이 생기면서
요구사항을 파악하고, 구현하고, 검증하는 과정을 하나의 프로세스로 정리할 수 있게 되었어요.

이번 글에서는 제가 만들어간 업무 프로세스와 그 안에서 AI를 활용하는 방식에 대해 정리해보려고 해요.

## 업무 프로세스

저는 업무를 아래 3단계로 나누어 진행하고 있어요.

### 1단계: 요구사항 파악

이 단계에서는 기존 레거시 화면의 동작을 하나씩 확인하면서, 코드 베이스에 녹아 있는 히스토리를 살펴봐요.
이때 단순히 화면만 보는 게 아니라, 왜 이렇게 동작하도록 구현했는지 맥락을 이해하는 데 집중하고 있어요.

특히, 이 과정에서는 코드를 직접 읽으면서 이해한 내용이 맞는지 AI와 함께 검증하고 있어요.
예를 들어 특정 로직의 분기 조건이 제가 이해한 의도와 일치하는지 확인하거나, 히스토리상 변경된 이유를 추론해보는 식이에요.

아무래도 주어진 한정적인 시간동안 모든 코드를 이해하는데 어려움이 있기 때문에,
코드 분석과 로직 검증에 AI를 적극적으로 활용하고 있어요.

### 2단계: 구현

두 번째 단계는 본격적인 구현이에요.

이 과정에서는 직접 만들어둔 3가지 커스텀 스킬을 활용하고 있어요.

| 스킬                 | 역할                                |
| -------------------- | ----------------------------------- |
| `api-researcher`     | API 엔드포인트 분석 및 명세서 작성  |
| `for-non-developers` | 비개발자도 이해할 수 있는 문서 생성 |
| `for-developers`     | 개발자 관점의 상세 구현 명세서 작성 |

구현 단계에서는 앞선 1단계에서 파악한 동작을 바탕으로 **인터페이스를 직접 설계**하고,
컴포넌트의 props 구조, API 호출 흐름, 상태 관리 방식 등을 먼저 정의한 뒤, `for-developers` 스킬을 통해 실제 코드 구현을 진행하고 있어요.

저는 인터페이스 설계와 플로우 검증은 직접 하고, 그 외 구현 작업에서는 AI를 **70~80%** 정도 활용하고 있어요.

### 3단계: 검증

세 번째 단계는 검증 단계에요.

검증 단계에서는 2단계에서 구현한 결과물이 1단계에서 확인했던 플로우와 일치하는지에 대해 비교하고,
동시에 `for-non-developers` 스킬로 생성한 비개발자 문서와도 비교하면서, 기능이 의도대로 동작하는지 다각도로 확인하고 있어요.

저는 주로 이 과정에서 스킬의 정확도를 개선하고는 하는데요.
현재 제가 판단하기에`for-developers` 스킬의 경우 약 **60~70%** 정도의 정확도로 느껴지는데, 이를 높이기 위해 `for-developers` 스킬 문서 및 여러 지침들을 지속적으로 업데이트하고 있어요.

비록 구현과정에서 AI를 많이 사용하지만, 그래도 최종적인 판단은 제가 하고 있어요.
그 이유는 설계에 대한 결정과 구현 결과에 대한 판단은 제가 해야 한다고 생각하기 때문이에요.

## 업무 프로세스를 돌아보며

위 3단계를 거치면 자연스럽게 세 종류의 명세서가 만들어지는데요.

1. **API 명세서** — `api-researcher` 스킬로 작성
2. **비개발자 명세서** — `for-non-developers` 스킬로 작성
3. **개발자 명세서** — `for-developers` 스킬로 작성

이 흐름에서 제가 가장 신경 쓰는 부분은, **처음 설계와 마지막 판단은 반드시 직접 한다**는 점이에요.
( 구현이 끝난 뒤에도, 이 결과물이 요구사항에 부합하는지 최종적으로 판단하는 건 사람이 해야 한다고 생각하기 때문이에요. 그에 따른 사이드이펙트 등도 사람이 잘 확인해야하고요. )

## 추가 고민: AI 시대의 PR 리뷰

최근 한 가지 더 고민하고 있는 주제가 있어요.

AI 덕분에 코드를 구현하는 속도는 눈에 띄게 빨라졌는데, 사람이 PR을 리뷰하는 속도는 예전과 크게 다르지 않다는 점이에요.
구현은 금방 끝나는데, 리뷰 대기 시간이 병목이 되는 상황이 종종 발생하거든요.

그래서 "리뷰어가 이 PR을 안심하고 승인할 수 있는 신뢰 장치가 있다면 어떨까?"라는 생각을 하게 되었어요.

제가 떠올린 방법은 **시나리오 기반 E2E 테스트**예요.
PR이 올라오면 해당 변경사항에 대한 E2E 테스트를 실행하고, 테스트 커버리지가 일정 수준 이상이면 리뷰어가 더 빠르게 판단할 수 있도록 돕는 거예요.

저는 이를 통해 이전 처럼 코드를 한 줄 한 줄 따라가지 않아도, 주요 시나리오가 검증되었다는 사실만으로도 리뷰 부담이 줄어들 수 있을 거라 기대하고 있어요.
하지만 아직 구체적인 테스트 설계가 갖춰지지 않은 상태라, 앞으로 이 부분을 시도해보려고 해요.

## 마무리

지금의 프로세스가 완벽하다고는 생각하지 않아요.
실제로 업무를 하면서 매번 조금씩 다듬고 있고, 앞으로도 계속 변할 것 같아요.
다만, 이렇게 프로세스를 의식적으로 만들어가는 과정 자체가 의미 있다고 느끼고 있어요.
